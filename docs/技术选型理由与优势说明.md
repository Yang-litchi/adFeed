## 1. 总览：为什么这样选

本项目技术选型的核心原则：

> **在满足课题要求的前提下，最小化团队的学习成本，最大化 2 周内的交付概率。**

具体来说，每项选择都经过以下三个维度的评估：

| 维度 | 权重 | 说明 |
|------|------|------|
| **学习成本** | 最高 | 团队 3 人均无 Java/Kotlin/Android 经验，技术栈必须"学得动" |
| **官方支持** | 高 | Google 官方推荐的库文档齐全、社区活跃、Bug 少 |
| **项目适配** | 中 | 技术方案要匹配 2 周项目规模，不过度设计 |

以下逐一解释每个选择的理由，每个选择都包含"我们的选择"和"为什么不选别的"。

## 2. 语言：Kotlin vs Java vs Flutter vs React Native

### 我们的选择：Kotlin

### 原因分析

#### 为什么不选 Java？

| 维度 | Kotlin | Java |
|------|--------|------|
| **代码简洁度** | ✅ 极简：`data class AdItem(...)` 一行搞定 POJO | ❌ 需要写 getter/setter/equals/hashCode/toString，数据类 100+ 行 |
| **null 安全** | ✅ 编译期 null 检查（`?` 语法），新手减少 NPE Bug | ❌ 运行时 NPE 是最大 Bug 来源之一，对于新手尤甚 |
| **协程** | ✅ 内置 `suspend` + `Flow`，异步代码像同步一样写 | ❌ 需要第三方库（RxJava）或手写 Thread/Callback |
| **扩展函数** | ✅ 如 `View.show()`、`String.toDate()`，代码更语义化 | ❌ 必须用工具类 `ViewUtils.show(view)` |
| **Android 官方态度** | ✅ "Kotlin-first"，所有新 Jetpack 库优先支持 Kotlin | ⚠️ 维护模式，新特性滞后 |
| **团队 C++ 背景** | ✅ Kotlin 的类型推导、sealed class 与 C++ enum class 概念相近 | ⚠️ Java 语法冗余较多 |

**关键差异对比**（同一个数据模型）：

```kotlin
// Kotlin — 1 行
data class AdItem(val id: String, val title: String, val type: MediaType)
```

```java
// Java — 30+ 行
public class AdItem {
    private String id;
    private String title;
    private MediaType type;
    public AdItem(String id, String title, MediaType type) { ... }
    public String getId() { return id; }
    public void setId(String id) { this.id = id; }
    public String getTitle() { return title; }
    // ... 还有 toString(), equals(), hashCode() 等
}
```

对于 2 周项目，代码量直接等于开发时间。Kotlin 的简洁意味着更快的学习和更少的 Bug。

#### 为什么不选 Flutter / React Native？

| 维度 | 原生 Android (Kotlin) | Flutter / React Native |
|------|----------------------|----------------------|
| **学习内容** | Kotlin + Android SDK（2 个概念域） | Dart + Flutter Widget 体系（3+ 个概念域） |
| **生态系统** | Android 原生 API 直接可用 | 需要桥接/Bridge，复杂功能需写原生插件 |
| **视频播放** | ExoPlayer 原生支持，稳定可靠 | 跨平台播放器兼容性问题多 |
| **课题要求** | 端侧明确要求 Android | - |
| **团队背景** | Kotlin 语法接近 C++/Python 理念 | Dart 是完全陌生的语言 |

对于明确要求 Android 端侧的课题，选择原生 Kotlin 是**唯一合理的技术路线**。

## 3. 数据策略：全本地 vs 网络请求

### 我们的选择：全本地数据（assets JSON + Room）

### 原因分析

| 维度 | 全本地 | 网络请求方案 |
|------|--------|-------------|
| **课题要求** | ✅ 课题未强制要求网络能力 | ⚠️ |
| **学习成本** | ✅ 无需学 Retrofit/OkHttp/HTTP 协议/API 设计 | ❌ 增加 ~2 天学习量 |
| **开发效率** | ✅ 数据即改即用，无需等待后端 | ❌ 连调前后端耗时 |
| **演示稳定性** | ✅ 100% 离线可用，不受网络/API 限制影响 | ❌ 依赖网络环境 |
| **功能完整性** | ✅ 所有核心功能（列表、详情、互动、统计）本地即可完整演示 | ⚠️ |
| **数据可控性** | ✅ 数据结构自定义、精确控制每条广告 | ❌ 依赖后端数据格式 |
| **AI 功能** | ✅ 预生成数据 + 规则匹配可实现 | ⚠️ 需处理 API Key/限流 |

**核心判断**：本课题的核心考核点是**客户端架构能力**（信息流性能、卡片动态化、状态管理、资源复用），而非网络通信能力。全本地数据方案让团队把有限的时间和注意力集中在真正考核的内容上。

### JSON + Room 双模策略的优势

```
assets JSON（只读）      Room DB（读写）
─────────────────      ───────────────
广告内容数据            互动状态（点赞/收藏）
频道列表                埋点事件记录
AI 预生成摘要/标签      曝光去重记录
                        AI 运行时缓存
```

- **优势1**：只读数据（广告）用 JSON，无需维护复杂的 ORM 映射，一个 Moshi 解析搞定
- **优势2**：读写数据（互动、埋点）用 Room，结构化查询 + 事务 + Flow 响应式查询
- **优势3**：两种数据源各司其职，不会出现"全用 Room 导致 JSON 字段变更需要数据库迁移" 或 "全用 JSON 导致互动状态查询需要每次全量扫描" 的问题

## 4. 架构：MVVM + 简化 Clean Architecture

### 我们的选择：MVVM + 简化 Clean Architecture

### 原因分析

```
为什么不是 MVC？
  → MVC 中 Activity/Fragment 既是 View 又是 Controller，代码臃肿难测试。
     3 人协作时 MVC 的代码耦合会导致频繁 Git 冲突。

为什么不是 MVP？
  → MVP 需要每个页面写 Presenter 接口 + View 接口，代码量大。
     对 2 周项目来说过度设计。

为什么不是 MVI？
  → MVI 的 Intent/Reducer/Effect 概念对新手不够直观。
     Google 官方的学习资料也偏向 MVVM。

为什么是 MVVM？
  → Google 官方推荐的 Android 架构
  → ViewModel + StateFlow 的模式与团队 C++ 背景的"数据驱动 UI"思想吻合
  → Sunflower 示例项目可作为直接参考
  → ViewModel 与 Fragment 生命周期解耦，配置变更（旋转）自动恢复
```

### Clean Architecture 的"简化"

我们采用的**不是**严格的 3 层 UseCase/Repository/DataSource 模式（那需要写大量 boilerplate），而是保留其**核心精神**：

```
保留的：                         简化的：
───────                         ───────
Repository 接口 → 依赖反转      跳过 UseCase 层（简单 CRUD 直接调 Repository）
分层依赖规则（向内依赖）         不强制 Data/Domain/Presentation 三层包结构
                                （小项目按模块分包更清晰）
```

**简化理由**：当广告数据的业务逻辑主要是"读取 → 过滤 → 展示"时，UseCase 层只是透明转发，不带来实际价值。保留 Repository 接口用于解耦，跳过 UseCase 直接调用，代码更少但架构精神不变。

## 5. 依赖注入：手工 DI vs Hilt vs Koin

### 我们的选择：手工 DI（AppContainer 模式）

### 对比分析

| 维度 | 手工 DI | Hilt | Koin |
|------|---------|------|------|
| **需学习的额外概念** | ✅ 0 个（纯 Kotlin class） | ❌ @HiltAndroidApp、@Module、@InstallIn、Component、Scope、@Binds、@Provides、@EntryPoint... ~10+ 个新概念 | ⚠️ module、factory、single、get()... ~5 个概念 |
| **编译速度** | ✅ 无额外开销 | ❌ APT 注解处理显著拖慢编译 | ✅ 快 |
| **代码透明度** | ✅ 依赖关系在 AppContainer 中一目了然，可断点调试 | ❌ 代码生成，看不到实际装配过程 | ⚠️ DSL 声明式，不够透明 |
| **错误时机** | 运行时（但因为是 AppContainer.kt 单文件，启动即触发） | 编译期（优势明显） | 运行时（可能到页面打开才发现） |
| **3 模块项目** | ✅ 刚好 | ❌ 杀鸡用牛刀 | ⚠️ 仍有一定过度 |
| **新手调试友好度** | ✅ 依赖关系像搭积木一样可见 | ❌ 出问题时"魔法"难以排查 | ⚠️ 需理解 Koin 容器机制 |

### 手工 DI 实现示意

```kotlin
// 仅仅一个文件 ~50 行，全部依赖关系一目了然
class AppContainer(context: Context) {
    // 基础设施
    private val database = Room.databaseBuilder(context, AppDatabase::class.java, "adfeed.db").build()
    private val localFileSource = LocalFileDataSource(context.assets)

    // Repository
    val adRepository = AdRepositoryImpl(localFileSource)
    val interactionRepository = InteractionRepositoryImpl(database.interactionDao())
    val analyticsRepository = AnalyticsRepositoryImpl(database.trackEventDao())

    // AI（可选）
    val aiSummaryService: IAISummaryService = if (ENABLE_AI) AISummaryServiceImpl(localFileSource)
                                              else NoOpSummaryService()

    // 播放器池
    val playerPool = PlayerPoolImpl(maxSize = 2)
}
```

**核心判断**：DI 框架的价值随项目规模增长。对于仅有 3 个功能模块、~10 个依赖项的项目，一个 AppContainer 类完全足够。将学习 Hilt/Koin 的时间（~1天）省下来投入核心功能开发是更优的策略。

## 6. UI：XML + ViewBinding vs Jetpack Compose

### 我们的选择：XML 布局 + ViewBinding

### 对比分析

| 维度 | XML + ViewBinding | Jetpack Compose |
|------|------------------|-----------------|
| **学习曲线** | ✅ 直观的 XML 树形结构，WYSIWYG 预览 | ❌ 声明式 UI 的心智模型完全不同 |
| **学习资料** | ✅ Android 10 年积累的教程、博客、StackOverflow | ⚠️ Compose 教程日益增多但仍少于 XML |
| **与现有代码对接** | ✅ RecyclerView、ExoPlayer PlayerView 原生 XML 支持 | ❌ 需要 Compose 互操作层 |
| **Google 官方示例** | ✅ Sunflower（MVVM+Room+Nav）全部 XML | ⚠️ Now in Android（Compose 但更复杂） |
| **团队 C++ 背景** | ✅ 类似 Qt 的 .ui 文件 | ❌ 声明式范式陌生 |
| **代码量** | ⚠️ 布局 + 逻辑分两个文件 | ✅ UI 和逻辑在同一个 Kotlin 文件 |

### 为什么不在这个项目中用 Compose？

1. **双重心智模型负担**：团队需要同时学习 Kotlin 语言 + Android 框架 + Compose 声明式范式，而 XML 布局是所见即所得，学习时不需要额外范式转换
2. **ExoPlayer 集成**：`PlayerView`（视频播放控件）是传统 View，在 Compose 中使用需要 `AndroidView` 包装，多一层复杂性
3. **RecyclerView 性能优化**：Compose 的 `LazyColumn` 虽然更简洁，但团队需要掌握的"列表性能"知识不同，而课题明确考核列表性能（Cell 复用、缓存池等），这些在传统 RecyclerView 中更直观

如果团队已有 Compose 经验，Compose 会是更好的选择。但考虑到团队的技能起点，**XML + ViewBinding 是更稳妥的决定**。

## 7. 图片加载：Coil vs Glide vs Picasso

### 我们的选择：Coil

### 对比分析

| 维度 | Coil | Glide | Picasso |
|------|------|-------|---------|
| 语言 | ✅ Kotlin 原生 | ❌ Java（Kotlin 调用需注意） | ❌ Java |
| 协程 | ✅ 一等公民 `suspend fun` | ⚠️ 需要 `RequestListener` 或额外适配 | ❌ 不支持 |
| API 简洁度 | ✅ `imageView.load(path)` | ⚠️ `Glide.with(ctx).load(path).into(view)` | ⚠️ 类似 Glide |
| 包体积 | ~2MB | ~4MB | ~3MB |
| 本地文件支持 | ✅ `file:///android_asset/` | ✅ | ✅ |
| 维护状态 | ✅ Square + Google 合力维护 | ✅ 活跃 | ⚠️ 不活跃（Square 已停止维护） |

**Coil 优势**：
- Kotlin 原生意味着扩展函数直接可用，代码更简洁
- 协程集成意味着可以在 ViewModel 中 `suspend` 预加载图片
- 轻量意味着更快的构建和更小的 APK

## 8. 数据库：Room vs 原生 SQLite vs Realm

### 我们的选择：Room

### 对比分析

| 维度 | Room | 原生 SQLite | Realm |
|------|------|------------|-------|
| 编译期检查 | ✅ SQL 编译期验证 | ❌ 运行时才能发现 SQL 错误 | ❌ 运行时 |
| Kotlin 协程 | ✅ Flow 原生支持 | ❌ 需要自己封装 | ⚠️ 需要适配 |
| 学习成本 | ✅ Google 官方文档配教程 | ⚠️ 需要手写 SQL | ❌ 独立生态 |
| 数据迁移 | ✅ Migration 机制 | ❌ 手写 SQL 升级脚本 | ⚠️ 自动但有坑 |
| 项目规模 | ✅ 正好 | ⚠️ 过多手动代码 | ❌ 过度 |

**关键理由**：Room 的编译期 SQL 校验对新手帮助最大——写错了字段名在编译时就报错，而不是运行时崩溃。这大幅降低了因 SQL 语法不熟导致的 Bug。

## 9. 视频播放：ExoPlayer vs ijkplayer vs 系统 MediaPlayer

### 我们的选择：ExoPlayer（Media3）

### 对比分析

| 维度 | ExoPlayer | ijkplayer | MediaPlayer |
|------|-----------|-----------|-------------|
| 官方支持 | ✅ Google 官方 | ❌ Bilibili 开源（维护不活跃） | ⚠️ 系统 API（功能弱） |
| 格式支持 | ✅ 全面（H.264/HEVC/VP9/AV1） | ✅ 全面 | ⚠️ 依赖设备 |
| 本地视频 | ✅ `RawResourceDataSource` 完美支持 | ✅ | ⚠️ 部分设备播放异常 |
| 自定义 UI | ✅ 完全可控 | ⚠️ 一般 | ❌ 受限 |
| 播放器复用 | ✅ 可创建多个实例池化管理 | ✅ | ❌ 单实例 |
| 崩溃率 | ✅ 低 | ⚠️ 部分设备兼容问题 | ⚠️ 不同厂商表现不一 |
| 学习资料 | ✅ 官方文档 + Codelab | ⚠️ 文档较少 | ⚠️ |

**关键理由**：ExoPlayer 对本地 `res/raw` 和 `assets` 视频文件有完善的支持，且提供了 `PlayerView` 控件可直接嵌入 XML 布局。作为一个需要播放器池（≤2实例）+ 外流/内流策略切换的功能，ExoPlayer 的灵活性是最佳匹配。

## 10. 列表：RecyclerView + 手动分页 vs Paging3

### 我们的选择：RecyclerView + DiffUtil + 手动分页

### 对比分析

| 维度 | RecyclerView + 手动分页 | Paging3 |
|------|------------------------|---------|
| 需学概念 | RecyclerView + Adapter（约 3 个概念） | PagingSource + PagingConfig + PagingData + LoadState + RemoteMediator（约 8 个概念） |
| 数据源 | 简单 Repository 调用 | 需要实现 `PagingSource` 子类 |
| 列表控制力 | ✅ 完全掌控（ViewHolder、动画、曝光计算） | ⚠️ 高度封装，定制需深入理解内部 |
| 与全本地数据适配 | ✅ 内存分页直接返回 List | ❌ Paging3 的设计前提是"数据源可能无限"，本地 JSON 固定数据过度设计 |
| 课程考核适配 | ✅ 题目考核"Cell 复用、缓存池"在 RecyclerView 中直接体现 | ❌ Paging3 封装掉了这些细节 |

**手动分页的核心代码**不到 20 行：

```kotlin
private suspend fun loadPage(page: Int) {
    val data = adRepository.getFeed(channelId, page, PAGE_SIZE)
    _feedState.update { it.copy(
        items = if (page == 1) data else it.items + data,
        hasMore = data.size >= PAGE_SIZE
    )}
}
```

对于数据总量已知（JSON 文件中的广告数）、不需要无限滚动的场景，手动分页比 Paging3 更简洁。

## 11. JSON 解析：Moshi vs Gson vs kotlinx.serialization

### 我们的选择：Moshi

### 对比分析

| 维度 | Moshi | Gson | kotlinx.serialization |
|------|-------|------|----------------------|
| Kotlin 支持 | ✅ 一等支持（data class 直接解析） | ⚠️ Java 优先，Kotlin null-safety 需额外处理 | ✅ Kotlin 原生 |
| 编译期检查 | ✅ 编译期生成 Adapter | ❌ 纯运行时反射 | ✅ 编译期 |
| 性能 | ✅ 快（编译期代码生成） | ⚠️ 较慢（运行时反射） | ✅ 最快 |
| 稳定性 | ✅ Square 维护，Android 社区标准 | ✅ Google 维护 | ⚠️ 相对较新 |
| 学习曲线 | ✅ 低（与 Gson 类似） | ✅ 最低（最常见） | ⚠️ 需 Gradle 插件配置 |

**选 Moshi 不选 Gson**：Moshi 对 Kotlin 的 null-safe 类型和默认参数处理更好。例如：

```kotlin
data class AdItem(val title: String, val aiSummary: String? = null)
// Gson 在 JSON 缺少 "aiSummary" 字段时可能抛异常
// Moshi 正确处理为 null
```

**不选 kotlinx.serialization**：虽然性能最好，但需要配置 Gradle 插件，学习成本略高于 Moshi。对于解析 ~100 条广告数据的场景，Moshi 的性能完全够用。

## 12. 设计模式选择理由

| 设计模式 | 为什么在这个项目中用它 | 不选择的后果 |
|---------|----------------------|-------------|
| **Repository** | 隔离数据源细节（JSON ↔ Room），ViewModel 不需要知道数据从哪来 | ViewModel 直接依赖 assets 读取，切换数据源需改全部 ViewModel |
| **Strategy（卡片渲染）** | 3 种卡片样式用 3 个策略类，新增样式不影响现有代码 | 所有卡片逻辑写在一个 Adapter 的 `onBindViewHolder` 里，200+ 行 switch |
| **Factory（ViewHolder）** | 根据 viewType 创建不同的 ViewHolder | `onCreateViewHolder` 中 if-else 判断，新增类型时改这里 |
| **Observer（SharedFlow）** | 跨页面状态同步（点赞/收藏）不需要手动传参 | 详情页点赞后返回列表，列表需要重新请求数据才能更新 |
| **Object Pool（播放器）** | 限制 ≤ 2 个 ExoPlayer 实例，防止内存爆炸 | 5 个视频卡片创建 5 个播放器 → OOM |
| **Template Method（BaseFragment）** | 统一 initViews → observeState → initListeners 生命周期流程 | 每个 Fragment 实现顺序不一致，代码风格混乱 |
| **Singleton** | Coil ImageLoader、Room Database 全局一份 | 多次初始化浪费资源 |

**共同目标**：每个设计模式都服务于**减少重复代码** + **降低模块耦合** + **让 3 人并行开发更顺利**。

## 13. 针对团队背景的优势总结

### 团队技能与选型的对应关系

| 选型 | 消除的学习负担 |

## 14. 学习成本评估

### 各团队成员需要学习的核心知识

```
Day 1 上午：Kotlin 核心（全员）
  □ 变量/类型/函数声明（15min）
  □ data class / sealed class / enum class（20min）
  □ null 安全（?, ?., ?:, !!, let/also/apply）（30min）
  □ 集合操作（map, filter, groupBy, first, find）（20min）
  □ 协程基础（suspend, launch, async, withContext）（30min）
  □ Flow 基础（StateFlow, SharedFlow, collect）（20min）

Day 1 下午 - Day 2：Android 核心（全员）
  □ Activity / Fragment 生命周期（20min）
  □ ViewModel / StateFlow 状态管理（30min）
  □ RecyclerView + Adapter + ViewHolder（40min）
  □ ViewBinding（15min）
  □ Room（Entity, DAO, Database）（30min）
  □ Navigation Component（20min）
  □ Coil 图片加载（10min）
  □ 项目骨架搭建实操（2h）

总计：约 1.5 天集中学习即可开始编码
```

### 技术难度分级

| 难度 | 技术 | 学习时间 | 备注 |
|------|------|---------|------|
| ⭐ 低 | Coil、Moshi、Timber、ViewBinding | < 1h 每个 | 简单的 API 调用 |
| ⭐⭐ 中 | Room、Navigation、RecyclerView、Coroutines | 2-3h 每个 | 需要理解概念，但有官方教程 |
| ⭐⭐⭐ 较高 | ExoPlayer（人员B 专项）、曝光追踪算法（人员C 专项） | 半天 | 需要深入理解，但仅各自负责 |

### 为什么不选择更简单的方案？

> **"最简单的方案不一定是最好的方案。"**

如果追求极致简单，可以用全静态布局 + 硬编码数据 + 无架构设计。但那样无法体现"技术方案设计能力"（占评分 35%）。当前方案在**可交付**和**有深度**之间找到平衡：

- 有 MVVM 分层 → 体现架构能力（评分维度1）
- 有 Repository 模式 + Room → 体现数据层设计（评分维度1）
- 有 DI 容器 + 接口解耦 → 体现解耦意识（评分维度1）
- 有 DiffUtil + RecycledViewPool + PlayerPool → 体现性能意识（评分维度1）
- 但每项都使用最低学习成本的实现 → 2 周可交付（评分维度2）

## 附录：技术选型决策树

```
需要 Android App
  │
  ├─ 语言？
  │   ├─ Kotlin ✅ （官方推荐、简洁、适合新手）
  │   └─ Java  ❌ （冗余、NPE风险）
  │
  ├─ 数据来源？
  │   ├─ 全本地 ✅ （无网络学习成本、演示稳定）
  │   └─ 网络    ❌ （额外学习+不稳定因素）
  │
  ├─ 架构？
  │   ├─ MVVM ✅ （官方推荐、资料多、上手快）
  │   ├─ MVI  ❌ （学习曲线陡）
  │   └─ MVP  ❌ （代码量大）
  │
  ├─ DI？
  │   ├─ 手工 DI ✅ （0 学习成本、3 模块够用）
  │   ├─ Hilt   ❌ （10+ 新概念）
  │   └─ Koin   ❌ （仍需学习 DSL）
  │
  ├─ UI？
  │   ├─ XML + ViewBinding ✅ （传统、直观、资料多）
  │   └─ Compose            ❌ （新范式、增加认知负荷）
  │
  ├─ 列表？
  │   ├─ RecyclerView + 手动分页 ✅ （直观、完全掌控）
  │   └─ Paging3                    ❌ （8+ 新概念、过度设计）
  │
  ├─ 图片？
  │   ├─ Coil   ✅ （Kotlin 原生、API 最简）
  │   └─ Glide  ❌ （Java 编写、API 冗长）
  │
  ├─ 数据库？
  │   ├─ Room      ✅ （编译期检查、Flow 原生支持）

  │
  ├─ JSON？
  │   ├─ Moshi ✅ （Kotlin 友好、null-safe）
  │   └─ Gson  ❌ （Java 优先、Kotlin 适配差）

  └─ 视频？
```
