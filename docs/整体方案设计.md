## 1. 项目概述

### 1.1 项目目标

构建一个 **AI 增强的单列广告信息流 Android App**，具备以下能力：

- **信息流展示**：单列流畅滚动、多样式广告卡片（大图/小图/视频）、频道切换、下拉刷新/上拉加载
- **资源复用**：Cell 复用、播放器实例池化、图片/视频缓存
- **数据与状态管理**：本地 JSON 文件 + Room 数据库双模数据源、跨页面状态同步、视频播放控制
- **AI 智能增强**（可选）：基于本地预生成数据或本地模型服务的广告摘要与智能标签、标签过滤、对话式搜索
- **埋点统计**：曝光/点击记录、互动统计、Room 持久化与可视化展示

### 1.2 核心约束

| 维度 | 约束 |
|------|------|
| 周期 | 2 周（6/1 – 6/14） |
| 端侧 | Android（Kotlin），IDE: Android Studio |
| 数据策略 | **全本地**：广告数据、用户行为数据、训练数据均来源于本地文件（assets/json）或本地 Room 数据库 |
| 网络 | **不涉及任何 HTTP 接口、云服务或远程网络请求** |
| 协作 | 3 人小组制，Git 管理 |
| 技术前提 | 三人均**无 Java/Kotlin 经验**（详见第 2 章） |
| 核心功能 | 必须完整实现 |
| 可选功能 | 加分项，需有降级方案 |
| 交付物 | GitHub 仓库 + 演示视频 + 设计文档（docs/） |

### 1.3 团队分工建议（结合技能背景）

### 2.4 因团队经验做出的设计简化

## 3. 总体架构设计

### 3.1 架构分层

采用 **MVVM + 简化 Clean Architecture 分层架构**，自底向上分为四层（**全本地数据，无网络层**）：

```
┌──────────────────────────────────────────────────────────┐
│                     Presentation 层                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────┐ │
│  │   Feed   │ │  Detail  │ │  Search  │ │   Stats    │ │
│  │ Fragment │ │ Fragment │ │ Fragment │ │  Fragment  │ │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └─────┬──────┘ │
│       │             │            │              │         │
│  ┌────┴─────┐ ┌────┴─────┐ ┌────┴─────┐ ┌─────┴──────┐ │
│  │ FeedVM   │ │ DetailVM │ │SearchVM  │ │ StatsVM   │ │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └─────┬──────┘ │
├───────┼─────────────┼────────────┼──────────────┼────────┤
│                     Domain 层（业务逻辑，纯 Kotlin）      │
│  ┌────┴─────────────┴────────────┴──────────────┴─────┐ │
│  │             UseCases / Repository 接口              │ │
│  │  GetFeedUseCase  GetDetailUseCase                  │ │
│  │  SearchAdUseCase  FilterByTagUseCase               │ │
│  │  ToggleLikeUseCase  TrackEventUseCase              │ │
│  └────────────────────────┬──────────────────────────┘ │
├────────────────────────────┼────────────────────────────┤
│                      Data 层（全本地数据实现）           │
│  ┌────────────────────────┴──────────────────────────┐ │
│  │        Repository Implementations                 │ │
│  │  ┌──────────────────┐  ┌──────────────────────┐  │ │
│  │  │ LocalFileSource  │  │    Room Database     │  │ │
│  │  │ (assets/*.json)  │  │ (interaction/cache/  │  │ │
│  │  │ 广告数据/频道数据  │  │  analytics)         │  │ │
│  │  └──────────────────┘  └──────────────────────┘  │ │
│  └──────────────────────────────────────────────────┘  │
├──────────────────────────────────────────────────────────┤
│                    Core 层（基础能力，纯本地）            │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────────┐ │
│  │  本地文件 │ │  图片    │ │  播放器  │ │  埋点统计  │ │
│  │ 读取工具  │ │ 加载器   │ │  池管理  │ │  引擎     │ │
│  │(Assets)  │ │ (Coil)   │ │(ExoPlayer)│ │          │ │
│  └──────────┘ └──────────┘ └──────────┘ └───────────┘ │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────────┐ │
│  │  手工 DI │ │  日志    │ │  导航    │ │  通用组件  │ │
│  │(Container)│ │ (Timber) │ │ (NavComp)│ │  (Utils)  │ │
│  └──────────┘ └──────────┘ └──────────┘ └───────────┘ │
└──────────────────────────────────────────────────────────┘

### 3.2 模块划分全景

项目划分为 **8 个模块**（Gradle 多模块工程）：

| 模块 | 类型 | 层级 | 职责 | 关键依赖 |
|------|------|------|------|---------|
| `app` | 应用壳 | 装配 | DI 容器初始化、路由导航、Application | 所有模块 |
| `core` | 基础库 | Core | 本地文件读取、图片加载(Coil)、日志(Timber)、Base类、通用工具 | 无（最底层） |
| `data` | 数据层 | Data | 数据模型定义、Repository实现、本地JSON加载、Room DAO、缓存策略 | `core` |
| `feed` | 功能模块 | Presentation | 信息流列表、卡片样式、Tab切换、下拉刷新/上拉加载 | `core`, `ai` |
| `detail` | 功能模块 | Presentation | 图文/视频详情、互动操作(点赞/收藏/分享)、状态同步 | `core`, `ai`, `player` |
| `player` | 功能模块 | Infrastructure | ExoPlayer封装、播放器池(≤2实例)、生命周期管理 | `core` |
| `ai` | 可选模块 | Domain+Data | AI数据加载/增强（AIDataService+AIAdRepository）、标签过滤、对话式搜索、AI结果缓存 | `core`, `data` |
| `analytics` | 功能模块 | Infrastructure | 曝光/点击追踪、互动统计、Room持久化、数据可视化 | `core`, `data` |

### 3.3 模块依赖关系

```
                    ┌─────────┐
                    │   app   │ (装配层：手工DI容器初始化、导航装配)
                    └────┬────┘
           ┌─────────────┼──────────────┐
           ▼             ▼              ▼
      ┌────────┐   ┌──────────┐   ┌──────────┐
      │  feed  │   │  detail  │   │analytics │
      └───┬────┘   └────┬─────┘   └────┬─────┘
          │             │              │
          └──────┬──────┘              │
                 ▼                     │
            ┌────────┐                 │
            │   ai   │ (AIAdRepository)│
            └───┬────┘                 │
                │                      │
                ▼                     │
            ┌────────┐                 │
            │  data  │ ◄───────────────┘
            └───┬────┘
                │
        ┌───────┼───────┐
        ▼       ▼       ▼
   ┌────────┐ ┌──────┐ ┌────────┐
   │  core  │ │player│ │(独立)  │
   └────────┘ └──────┘ └────────┘
```

**关键依赖原则**：
- `core` 不依赖任何模块（最底层，只依赖 Android SDK 和第三方库）
- `data` 依赖 `core`（使用文件读取、日志、调度器工具），提供 `AdRepository` 实现
- `ai` 依赖 `data`（`AdRepository` 接口）和 `core`，通过 `AIAdRepository` 对上层暴露增强后的数据
- `feed` / `detail` 依赖 `ai`（`AIAdRepository` 接口），不直接依赖 `data`
- `analytics` 依赖 `data` 和 `core`（埋点数据独立于广告增强流程）
- `ai` 模块中 AI 数据缺失时自动降级（description 截断兜底），不影响核心功能
- `app` 是唯一汇集点：手工创建各模块实例并注入

## 4. 架构设计原则与设计模式

### 4.1 SOLID 原则应用

| 原则 | 应用方式 | 收益 |
|------|---------|------|
| **S** - 单一职责 | `player` 只负责播放；`analytics` 只负责埋点；Repository 只做数据协调 | 模块独立开发、独立测试 |
| **O** - 开闭原则 | `CardRenderStrategy` 接口 + 策略模式，新增卡片类型只需加新策略类 | 卡片类型可独立扩展 |
| **L** - 里氏替换 | BaseFragment/BaseViewModel 统一基类契约；Repository 的本地文件实现与内存实现可互换 | 测试与真实运行环境无痛切换 |
| **I** - 接口隔离 | Repository 按业务拆分为 `FeedRepository`、`InteractionRepository`、`SearchRepository` 等小接口 | 每个页面只依赖自己需要的接口 |
| **D** - 依赖倒置 | ViewModel 依赖 Domain 层的 Repository 接口，不依赖 Data 层的具体实现 | 单元测试可 Mock；数据源可替换 |

### 4.2 设计模式应用

| 设计模式 | 应用场景 | 说明 |
|---------|---------|------|
| **Repository** | 数据访问抽象层 | 统一封装本地 JSON 文件读取 + Room 数据库访问，对外暴露单一接口 |
| **Strategy（策略）** | 卡片渲染 | 每种卡片样式（大图/小图/视频）对应一个 `CardRenderStrategy` 实现 |
| **Factory（工厂）** | ViewHolder 创建 | `CardViewHolderFactory` 根据 `viewType` 创建对应 ViewHolder |
| **Observer（观察者）** | 跨页面状态同步 | `SharedFlow` / `StateFlow` 广播互动事件（点赞/收藏） |
| **Decorator（装饰器）** | AI 内容增强 | `AIEnhanceDecorator` 在原始 AdItem 上附加摘要和标签 |
| **Object Pool（对象池）** | 播放器实例复用 | `PlayerPool` 限制最大 2 个播放器实例，空闲回收 |
| **Adapter** | RecyclerView 适配 | ViewHolder 复用 + DiffUtil 增量更新 |
| **Singleton** | 图片加载/缓存/数据库 | Coil ImageLoader、CacheManager、RoomDatabase 全局唯一 |
| **Chain of Responsibility** | 埋点事件处理链 | 曝光检测 → 去重 → 持久化 → 统计更新 |
| **Template Method** | Base 类抽象 | `BaseFragment.initViews() → observeState() → initListeners()` 模板 |
| **Simple Factory** | 手工 DI 容器 | `AppContainer` 集中创建并注入各模块依赖 |

### 4.3 分层设计理念

```
Presentation ──→ Domain ──→ Data
   (View+VM)     (UseCase)   (Repo+Source)
                    ↑              ↑
                接口定义        接口实现
              (Domain层)      (Data层：本地文件 + Room)

依赖方向：Presentation → Domain ← Data
            （Domain 是中心，Data 实现 Domain 接口，实现依赖反转）
```

**核心规则**：
- **Domain 层不依赖任何框架**（纯 Kotlin），业务逻辑可独立测试
- **Data 层数据全部来自本地**：`assets/*.json`（广告数据）+ Room（互动/缓存/埋点）
- **Presentation 层只感知 UseCase/Repository 接口**，完全不知道数据存取细节

## 5. 技术选型与对比

### 5.1 技术栈总览

| 类别 | 技术选型 | 选型理由 |
|------|---------|---------|
| **语言** | Kotlin 1.9+ | Google 官方推荐；null 安全减少 NPE Bug；data class/sealed class 简洁表达数据模型 |
| **架构模式** | MVVM + 简化 Clean Architecture | 官方推荐、学习资料丰富、分层清晰 |
| **DI** | **手工 DI**（AppContainer 模式） | 省去 Hilt/Koin 学习成本，3 人小项目完全可控，代码透明可调试 |
| **本地数据加载** | Android `AssetManager` + `Moshi` JSON 解析 | 广告数据预置在 `assets/ad_data.json`，标准 Android API 无需额外库 |
| **图片加载** | **Coil** 2.5+ | Kotlin 原生、协程集成、API 简洁（`imageView.load(url)`）、面向本地文件也可用 |
| **视频播放** | **ExoPlayer (Media3)** 1.2+ | Google 官方、播放本地视频文件稳定、自定义控制 UI 方便 |
| **本地数据库** | **Room** 2.6+ | Google 官方、编译期 SQL 校验、Flow 响应式查询 |
| **导航** | **Navigation Component** 2.7+ | Jetpack 官方、SafeArgs 类型安全 |
| **异步** | **Kotlin Coroutines + Flow** | 结构化并发、生命周期感知、Flow 天然适配 Room |
| **列表** | **RecyclerView** + **DiffUtil** | ViewHolder 复用 + 高效增量更新 |
| **JSON 解析** | **Moshi** 1.15+ | Kotlin 优先、编译期生成适配器 |
| **日志** | **Timber** | 轻量、比 Log 更好用的封装 |
| **视频资源** | `res/raw/` 或 `assets/video/` | 本地视频文件 |
| **图片资源** | `assets/images/` | 本地图片文件，由 Coil 从 assets 加载 |
| **构建** | Gradle + Version Catalog | 统一依赖版本 |

### 5.2 关键选型对比

#### 5.2.1 DI：手工 DI vs Hilt vs Koin

| 维度 | 手工 DI (AppContainer) | Hilt | Koin |
|------|-----------------------|------|------|
| **学习成本** | ✅ 极低（纯 Kotlin 代码） | ❌ 高（注解处理、Component、Scope 概念） | ⚠️ 中（DSL语法、Module定义） |
| **实现代码量** | ⚠️ 需手写工厂方法 | ✅ 注解自动生成 | ✅ DSL 声明式 |
| **编译速度** | ✅ 最快（无注解处理） | ❌ 慢（APT 注解处理） | ✅ 快（无注解处理） |
| **运行时性能** | ✅ 最佳（无反射） | ✅ 最佳（无反射） | ⚠️ 有少量反射 |
| **错误发现** | ⚠️ 运行时 | ✅ 编译期 | ❌ 运行时 |
| **3模块项目适配** | ✅ 完全够用 | ⚠️ 过度设计 | ⚠️ 过度设计 |
| **2周交付+新手** | ✅ **最佳选择** | ❌ 学习成本太高 | ❌ 仍需学习 Koin DSL |

**最终选型：手工 DI（AppContainer 模式）** — 代码透明、无学习成本、3 模块规模足够。详见《[技术选型理由与优势说明](./技术选型理由与优势说明.md)》

#### 5.2.2 图片加载：Coil vs Glide

| 维度 | Coil | Glide |
|------|------|------|
| 语言 | ✅ Kotlin 原生 | ❌ Java 编写 |
| 协程支持 | ✅ 一等公民 | ⚠️ 需额外适配 |
| API 简洁度 | ✅ `imageView.load(url)` | ⚠️ 链式调用较长 |
| 本地 assets 支持 | ✅ 支持 `file:///android_asset/` | ✅ 支持 |
| 包体积 | ~2MB | ~4MB |

**最终选型：Coil** — Kotlin 原生、API 简洁、新手友好。

#### 5.2.3 数据源：本地 JSON 文件 vs Room 数据库

| 维度 | assets/*.json | Room 数据库 |
|------|--------------|------------|
| 适用数据 | 只读型：广告内容、频道列表 | 读写型：互动状态、埋点、AI 缓存 |
| 查询能力 | 全量加载后在内存中过滤 | SQL 查询，支持条件过滤、排序、聚合 |
| 持久化 | APK 内只读（升级时可替换） | 可读写，App 重启数据保持 |
| 数据变更 | 需更新 APK | 运行时随时写入 |

**选型结论**：**两者组合使用** — 广告数据/频道数据用 JSON 文件（只读、预置），互动状态/埋点/AI缓存用 Room（读写、持久化）。

#### 5.2.4 架构模式：MVVM vs MVI

| 维度 | MVVM | MVI |
|------|------|------|
| 学习成本 | ✅ 低（Android 官方教程） | ❌ 中（Intent/Reducer 概念） |
| Google 官方支持 | ✅ 官方推荐+Sunflower 示例 | ⚠️ 社区驱动 |
| 状态管理 | 多 StateFlow（分散但直观） | 单一 State（聚合但复杂） |
| 2周+新手适配 | ✅ 最佳选择 | ❌ 额外学习负担 |

**最终选型：MVVM** — 官方文档齐全、新手可快速上手、2 周项目规模最佳。

### 6.3 降级方案

所有降级由 `AIAdRepositoryImpl` 内部统一处理，feed/detail 无感知：

| 可选功能 | 降级行为 | 用户感知 |
|---------|---------|---------|
| AI 摘要展示 | AI 数据缺失时 summary = description 截断 50 字 + "..." | 卡片始终展示摘要，只是来源不同 |
| AI 标签展示 | AI 数据缺失时 tags = emptyList，标签区域不展示 | 无标签时卡片自动隐藏标签栏 |

| 标签点击过滤 | 基于 `AIEnhancedAd.tags` 做客户端过滤，无 AI 数据时过滤功能无意义但不报错 | 有 AI 数据时可筛选，无 AI 数据时标签过滤返回空列表 |

                              └────────┬─────────┘
                                       │
                        ┌──────────────┼──────────────┐
                        ▼              ▼              ▼
                  ┌───────────┐ ┌───────────┐ ┌────────────┐

### 7.2 跨页面状态同步（点赞/收藏）

**方案：SharedFlow 事件总线 + Room 持久化**

```
              feed 模块                     detail 模块
         ┌──────────────┐              ┌──────────────┐
         │ FeedViewModel │              │DetailViewModel│
         │              │              │              │
         │ interaction  │──SharedFlow─→│ interaction  │
         │ StateFlow    │   事件广播    │ StateFlow    │
         └──────┬───────┘              └──────┬───────┘
                │                             │
                ▼                             ▼
         ┌─────────────────────────────────────────┐
         │        InteractionRepository            │
         │   ┌───────────┐  ┌──────────────────┐  │
         │   │ Room 持久化│  │ SharingFlow 广播  │  │
         │   │(interaction│  │ (同进程内存同步)   │  │

         └─────────────────────────────────────────┘
```

**同步流程**：
1. 用户在详情页点赞
2. `DetailViewModel` → `InteractionRepository.toggleLike(adId)`

4. Repository 通过 `SharedFlow` 广播 `InteractionEvent(adId, LIKE, newState)`

6. 用户返回信息流，对应用卡片的点赞状态已正确

### 7.3 全本地数据生命周期管理

| 数据类型 | 存储位置 | 生命周期 | 淘汰策略 |
|---------|---------|---------|---------|
| 广告列表原始数据 | `assets/ad_data.json` | APK 级，随安装存在 | 不淘汰，只读 |
| 频道列表数据 | `assets/channels.json` | APK 级，随安装存在 | 不淘汰，只读 |

| 广告详情缓存 | 内存 LRU Cache | 进程存活 | LRU ≤ 50 条 |

| AI 摘要/标签（预生成） | `assets/ai_data.json` | APK 级 | 不淘汰，只读 |
| AI 缓存（运行中生成） | Room `ai_cache` 表 | 24h TTL | 定时清理 + 内容哈希失效 |
| 埋点事件数据 | Room `track_events` 表 | App 级持久化 | 用户主动清除 |
| 曝光记录（去重用） | 内存 Set + Room | 进程存活 + 持久化 | 按 sessionId 去重 |

private suspend fun loadPage(page: Int) {

### 9.4 AI 输出约束与缓存策略

**广告数据与 AI 数据彻底分离**：`AdItem` 不含任何 AI 字段，`ad_data.json` 不嵌入 AI 结果。AI 分析结果独立存储在 `assets/ai_data.json`。

| | 开发期（优先） | 完成期（加分） |
|------|-------------|-------------|
| 数据来源 | `LocalAIDataService` 读 `assets/ai_data.json` | `RemoteAIDataService` 调大模型 API + Room 缓存 |
| 实现 | 预生成 JSON，Moshi 解析为 `Map<String, AIData>` | 网络请求 → 解析 → 写 Room 缓存 → 返回 |
| 稳定性 | 100% 稳定，离线可用 | 依赖网络，有缓存兜底 |
| AI 数据缺失 | `aiDataService.loadAIData()` 返回 `emptyMap()` | API 失败 → 查 Room 缓存 → 缓存未命中 → 降级 |
| 切换方式 | — | AppContainer 中一行代码切换 `AIDataService` 实现 |

      {"name": "学生党", "category": "AUDIENCE"},
      {"name": "户外", "category": "SCENE"}

```

| 场景 | 摘要 | 标签 |
|------|------|------|
| `ai_data.json` 缺失 | description 截断 50 字 + "..." | emptyList（不展示） |
| 某广告无对应 AI 数据 | 同上 | 同上 |
| 大模型 API 失败（完成期） | 同上 | 同上 |

## 10. 边界条件与风险管理

### 10.1 边界条件清单

| 场景 | 边界条件 | 处理策略 |
|------|---------|---------|
| **空数据** | 某频道广告数为 0 | 展示空态页面（插图 + "暂无内容"） |
| **本地文件加载失败** | assets JSON 解析异常/文件缺失 | 展示错误态 + "数据加载失败，请重启 App" |
| **图片文件不存在** | 本地 assets 图片路径无效 | Coil 展示 placeholder 占位图 |
| **视频解码失败** | 不支持的本地视频编码 | 降级为封面图 + 错误提示 |
| **快速切换 Tab** | 用户快速切换频道 | 旧协程自动 `cancel()` |
| **快速滚动** | 惯性滑动导致大量图片解码 | 仅加载可见 + 预取区域图片 |
| **详情返回** | 信息流被系统回收 | `SavedStateHandle` 保存频道 + 滚动位置 |
| **内存不足** | 大量图片缓存 | `onTrimMemory()` 时释放 Coil 内存缓存 |
| **横竖屏切换** | 详情页旋转 | 保存播放进度到 `SavedStateHandle`，重建后恢复 |
| **重复点赞** | 快速双击 | 前端防抖 500ms |
| **并发修改** | 多线程同时写 Room | Room `@Transaction` 事务保障 |
| **Room 版本升级** | 数据库 schema 变更 | 开发阶段用 `fallbackToDestructiveMigration()` |

### 10.2 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| **Kotlin/Android 学习曲线陡峭** | 高 | 极高 | Day1-2 集中速学；只学项目用到的部分；看不懂的先模仿 |
| **ExoPlayer 使用复杂** | 中 | 高 | 简化使用：只封装 play/pause/seek/mute，不做高级自定义 |
| **Room 数据库上手慢** | 中 | 中 | 从最简单的单表 CRUD 开始，逐步加查询 |
| **RecyclerView 复用 Bug** | 中 | 中 | `onViewRecycled()` 中释放播放器；`onViewAttachedToWindow()` 重新绑定 |
| **2 周时间不够** | 中 | 高 | MoSCoW 优先级排序；核心功能先完成；可选功能按接口预留 |
| **Git 冲突** | 低 | 中 | 模块化目录结构；每人改不同文件；每日 rebase main |
| **Android 模拟器/设备兼容** | 低 | 中 | 统一用 API 28+ 模拟器；ExoPlayer 官方兼容性好 |
