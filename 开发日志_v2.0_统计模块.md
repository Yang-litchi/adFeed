## 本次迭代总结

本期迭代完成了**广告统计系统**的全链路建设，包括：统计数据模型设计、内存仓库实现、统计页面UI及图表可视化、曝光检测系统（含可见性判定、计时逻辑、会话级去重）、以及跨页面（Feed列表 / 详情页 / AI搜索入口）的曝光与点击事件统一采集。

同时修复了三个影响统计准确性的关键Bug：**AI悬浮球点击未计入Click**、**详情页缺少曝光追踪**、**HorizontalPager共享LazyListState导致Feed曝光统计全部失效**。最终实现了主页面三频道独立曝光统计、AI搜索结果统计、详情页统计的全面正常运作。

## 功能更新

### 1. 统计模块整体架构

**新增数据模型：**
- `StatisticEvent` — 统计事件记录，承载广告ID、标签列表、事件类型（曝光/点击）、时间戳
- `StatisticSummary` — 单广告统计摘要，包含总曝光量、总点击量、CTR（点击率）
- `TagStatistic` — 同Tag广告对比数据，支持横向曝光/点击对比
- `TrendData` — 单日趋势数据点，用于7天趋势折线图

**新增仓库层：**
- `StatisticsRepository` 接口 — 定义统计数据的标准访问契约（记录曝光/点击、查询广告统计/同Tag对比/趋势数据）
- `FakeStatisticsRepository` 内存实现 — 使用线程安全数据结构（`CopyOnWriteArrayList` + `ConcurrentHashMap`），确保FeedViewModel（写入端）与StatisticsViewModel（读取端）之间的数据一致性。后续替换为Room实现时无需修改任何业务层代码

**新增ViewModel：**
- `StatisticsViewModel` — 统计页面专属ViewModel，支持三路数据并行加载（广告摘要、同Tag对比、趋势数据），提供加载中/错误/重试状态管理

**新增UI页面和组件：**
- `StatisticsScreen` — 统计页面，展示三个数据区块：广告统计摘要卡片、同Tag对比柱状图、7天趋势折线图，支持错误重试
- `BarChart` — 水平柱状图组件（Compose Canvas实现），用于同Tag广告曝光量直观对比
- `LineChart` — 折线图组件（Compose Canvas实现），支持双线（曝光/点击）+ 数据点标记 + 图例，用于7天趋势展示

**影响范围：** 新增8个源文件（4个数据模型 + 1个接口 + 1个实现 + 1个ViewModel + 1个Screen + 2个图表组件），修改2个文件（MainActivity导航、FeedScreen曝光统计接入点）

### 2. 曝光检测系统（ExposureDetector + ExposureTracker）

**设计原则：** 曝光判定逻辑从UI页面中完全解耦，形成独立的可复用模块。

**曝光判定规则（广告行业标准）：**
1. 广告可见面积 ≥ 50% → 进入候选状态
2. 在满足可见性条件下**连续展示 ≥ 1秒** → 计为1次有效曝光
3. 计时期间可见面积下降到阈值以下 → 自动取消本次计时，重新进入候选时重新计时
4. 同一广告在单次会话（App启动→关闭）内仅允许1次有效曝光

**新增组件：**
- `ExposureTracker` — 会话级曝光状态追踪器（object单例），维护当前会话中已完成有效曝光的广告ID集合，跨页面/跨ViewModel全局唯一
- `ExposureDetector` — 可复用Compose组件，接收广告ID + 可见面积比例 + 回调，内部通过`LaunchedEffect` key设计自动管理计时/取消逻辑

**可见面积计算：** 基于`LazyListState.layoutInfo`实时计算每个列表项的可见像素占比，为`ExposureDetector`提供精确的`visibleFraction`输入

**影响范围：** 新增2个独立模块文件，修改FeedScreen接入曝光检测

### 3. 滑动导航（SwipeNavigable）

**功能：** Compose Modifier扩展，检测水平滑动手势，累积位移超过阈值（默认120dp）后触发导航回调

**设计特点：**
- 与垂直滚动（LazyColumn、verticalScroll）互不冲突 — 仅响应水平为主的手势
- 手势识别（修饰符）与页面导航（回调参数）完全解耦
- 支持自定义左右滑动方向分别配置回调

**应用场景：** 详情页右→左滑动进入统计页，统计页左→右滑动返回详情页

### 4. Feed列表曝光统计接入

在Feed列表的每个广告卡片渲染时，通过`ExposureDetector`组件自动采集曝光事件，触发条件满足后调用`FeedViewModel.recordExposure()`，该操作同步更新UI状态（内存计数器）并异步写入统计仓库。

### 5. 点击统计双入口覆盖

- **Feed卡片点击：** 通过`onCardClick`回调 → `recordClick()` → 导航
- **AI悬浮球点击：** 通过`AiFloatingBall.onAdClick`回调 → `recordClick()` → 导航

两个入口行为统一，确保无论用户从哪个入口点击广告，Click事件均被正确记录。

## Bug修复

### Bug #1：AI悬浮球搜索结果点击未记录Click事件

**问题描述：** 用户通过AI悬浮球搜索结果点击广告并进入详情页后，该广告的点击数（Click）未被统计。

**根因分析：** Feed卡片点击在`onCardClick`回调中先调用了`viewModel.recordClick(ad.id)`再触发导航，但AI悬浮球的`onAdClick`回调直接使用了MainActivity传入的原始回调（仅做导航`navController.navigate("detail/${ad.id}")`），缺少`recordClick`调用。

**修复内容（1处修改）：** 在`FeedScreen.kt`中，将`AiFloatingBall(onAdClick = onAdClick)`改为包装回调，先执行`viewModel.recordClick(ad.id)`再执行原始导航回调。

**影响范围：** AI悬浮球搜索结果中的广告链接点击

### Bug #2：详情页缺少曝光追踪

**问题描述：** 用户进入广告详情页（无论从Feed卡片、AI搜索结果还是其他入口）后，该广告的曝光数（Exposure）完全不被统计。Feed列表有`ExposureDetector`但其仅在列表可见时生效，详情页是独立页面，此前完全没有曝光采集逻辑。

**根因分析：** `DetailScreen`既没有使用`ExposureDetector`，也没有调用`viewModel.recordExposure()`。对于从AI搜索结果直接进入详情页的场景（广告从未在Feed中出现），Exposure事件完全缺失。

**修复内容：** 在`DetailScreen`的内容渲染处添加`ExposureDetector`，参数`visibleFraction = 1f`（详情页内容默认100%可见），使用与Feed列表完全一致的曝光判定规则（1秒计时 + `ExposureTracker`会话去重）。

**去重保障：** `ExposureTracker`为全局单例，跨Feed列表和详情页共享已曝光集合。如果某广告已在Feed中完成曝光，进入详情页后`ExposureDetector`检测到`isExposed=true`自动跳过；如果从AI搜索直接进入（Feed中未曝光），则在详情页正常触发曝光。

**影响范围：** 所有进入详情页的入口（Feed卡片、AI搜索结果、外部链接等）

### Bug #3：主页面Feed列表曝光统计全部失效

**问题描述：** 用户在Feed列表中滚动浏览广告，广告可见面积≥50%且停留数秒后，曝光计数无任何变化。主页面疑似所有曝光都没有被统计。

**根因分析：** 这是一个架构层面的严重Bug：

- `HorizontalPager`有3个页面（精选/电商/本地），每个页面有一个`LazyColumn`
- 三个`LazyColumn`共享同一个`LazyListState`实例（在`FeedScreen`顶层创建）
- `HorizontalPager`默认**预组合相邻页面**，离屏页面的`LazyColumn`执行布局时会**覆写共享`LazyListState`的`layoutInfo`**
- 当离屏页面后执行布局且其viewport为0或数据为空时，基于`layoutInfo`计算的`visibleFractionMap`被覆写为空Map或全0数据
- 连锁反应：`ExposureDetector`收到`visibleFraction = 0` → `meetsThreshold = false` → 曝光条件永不满足 → **所有广告的曝光全部无法触发**

**修复方案：** 将`rememberLazyListState()`和`visibleFractionMap`从`FeedScreen`顶层作用域移入`HorizontalPager`每页的内容lambda内部，使每个频道的`LazyColumn`拥有独立的`LazyListState`和可见性计算。

**修复前后架构对比：**
```
修复前（错误）:                          修复后（正确）:
FeedScreen                              FeedScreen
  listState ← (1个, 3页共享, 互相覆盖)     (无共享状态)
  visibleFractionMap ← (共享, 数据错乱)     └─ HorizontalPager
  └─ HorizontalPager                          ├─ Page 0: listState₀, fracMap₀
       ├─ Page 0: LazyColumn(state)                LazyColumn(state₀)
       ├─ Page 1: LazyColumn(state)           ├─ Page 1: listState₁, fracMap₁
       └─ Page 2: LazyColumn(state)                LazyColumn(state₁)
                                               └─ Page 2: listState₂, fracMap₂
                                                     LazyColumn(state₂)
```

**影响范围：** Feed列表所有三个频道（精选/电商/本地）的曝光统计，点击统计不受影响

## 代码优化

### 1. 曝光/点击事件职责分离设计

明确了Click（点击）和Exposure（曝光）是两个独立且职责不同的统计事件，在代码层面彻底分离：
- **Click** — 在用户交互点即时记录（Feed卡片`onCardClick`、AI悬浮球`onAdClick`），由用户触摸操作驱动
- **Exposure** — 由`ExposureDetector`在可见性条件满足后延迟触发（≥50%可见 + 持续1秒），由可见性状态变化驱动
- 两者通过不同的触发路径、不同的回调、不同的时机记录，互不混淆

### 2. 统计仓库接口抽象（面向接口编程）

`StatisticsRepository`接口定义标准数据访问契约，当前使用`FakeStatisticsRepository`内存实现。后续替换为Room数据库实现时，仅需新增一个`StatisticsRepository`的实现类，**无需修改任何业务层代码**（ViewModel、UI层零改动）。

### 3. ExposureDetector模块化解耦

曝光检测逻辑封装为独立Compose组件，与具体页面、ViewModel、数据源完全解耦。任何页面只需引入`ExposureDetector`并传入`adId + visibleFraction + onExposed`回调即可复用，无需关心内部计时/去重/取消逻辑。

### 4. 数据加载并行化

`StatisticsViewModel.loadStatistics()`使用`coroutineScope.async`并行加载三份统计数据（广告摘要、同Tag对比、趋势数据），减少用户等待时间。
